# COMMERCIALIZATION_ANALYSIS

## 1. 목적

이 문서는 현재 `Nightseed Survivor`를 플레이 가능한 MVP에서 상용 출시 후보로 끌어올리기 위해 필요한 개선 방향을 정리한다.

분석 기준일은 2026-05-14이며, 아래 자료를 함께 확인했다.

- 로컬 구현: `godot/scripts/`, `godot/scenes/`, `godot/data/`, `play_store/` 스크린샷
- 프로젝트 기준 문서: `docs/GAME_SPEC.md`, `docs/IMPROVEMENT_STRATEGY.md`, `docs/ASSET_GUIDE.md`, `docs/ARCHITECTURE.md`, `docs/BALANCE.md`
- 시장/플랫폼 참고:
  - Google Play Games Level Up guidelines: https://developer.android.com/games/guidelines
  - Google Play ads policy: https://support.google.com/googleplay/android-developer/answer/9857753
  - Google Play game services statistics: https://support.google.com/googleplay/android-developer/answer/3423625
  - GameRefinery mobile game market review, April 2026: https://www.gamerefinery.com/mobile-game-market-review-april-2026/
  - Survivors-like 장르 개요: https://en.wikipedia.org/wiki/Vampire_Survivors%E2%80%93like

## 2. 결론 요약

현재 게임은 "플레이 가능한 MVP" 수준은 넘었다. 무기 5종, 캐릭터 5종, 스테이지 5종, 난이도, 업적, 스토리 배너, 로컬 저장, 리더보드 연동, 스토어 자산까지 들어와 있어 기본 제품 골격은 있다.

하지만 상용화 관점에서는 아직 "돈을 받거나 스토어에서 적극 홍보하기 좋은 완성품"으로 보기는 어렵다. 가장 큰 문제는 콘텐츠 양보다 완성도 계층이다.

우선순위는 다음 순서가 적절하다.

1. 디자인/아트 일관성 정리
2. 첫 3분 재미와 빌드 선택 체감 강화
3. 밸런스/콘텐츠 데이터를 코드 밖으로 분리
4. 모바일 UX와 성능 안정화
5. 저장/업적/리더보드/스토어 정책 품질 정리
6. 출시 전 QA 루틴과 지표 기반 튜닝 체계 마련

즉, 다음 단계는 기능을 더 많이 붙이는 것이 아니라 "현재 있는 기능을 제품처럼 느끼게 만드는 작업"이다.

## 3. 현재 상태 판단

### 강점

- 기본 조작이 단순하다. 이동 중심, 자동 공격 구조가 장르와 모바일에 잘 맞는다.
- 캐릭터, 스테이지, 난이도, 상점, 업적, 용어집이 이미 있어 반복 플레이의 틀이 있다.
- `stages.json` 기반 웨이브는 향후 밸런스 조정이 쉬운 방향이다.
- 스토리 배너와 Codex는 MVP 범위를 크게 넘지 않으면서 차별화 여지를 만든다.
- Play Games Services 연동과 스토어 자산이 있어 출시 파이프라인의 뼈대가 있다.
- Kenney CC0 애셋과 Pretendard 폰트를 사용해 라이선스 리스크가 낮다.

### 약점

- 화면 전체의 아트 방향이 아직 placeholder와 실제 픽셀 애셋 사이에 걸쳐 있다.
- UI 버튼, HUD, 레벨업 카드가 기능은 하지만 상용 게임의 밀도/리듬/시각 위계가 부족하다.
- 무기/패시브 선택 설명이 일부 일반화되어 있어 선택의 차이가 덜 느껴진다.
- `Characters.gd`, `LevelUpUI.gd`, `WeaponManager.gd`, `GameData.gd` 등에 주요 수치와 설명 구조가 남아 있어 빠른 밸런스 반복에 한계가 있다.
- 플레이 데이터 수집 수단이 제한적이다. 외부 분석 SDK를 넣지 않는 정책은 유지하되, 내부 QA용 로컬 리포트는 필요하다.
- `.agent/tasks.md`와 README/실제 구현 상태가 서로 맞지 않는다. 작업 관리 문서가 뒤처져 다음 우선순위 판단을 흐린다.

## 4. 디자인 개선 방향

### 4.1 아트 디렉션

현재 스크린샷은 큰 캐릭터 픽셀 스프라이트, 단색 배경, 단순 UI 패널이 섞여 있다. 플레이는 읽히지만 상용 제품의 첫인상은 약하다.

권장 방향:

- "어두운 밤 + 창백한 달빛 + 기억의 파편"을 핵심 시각 언어로 고정한다.
- 배경색을 스테이지별 단색이 아니라 2~3단계 명도 레이어로 만든다.
- 캐릭터/적은 지금보다 작고 명확한 실루엣으로 정리한다.
- 플레이어, 적, 픽업, 위험 영역, 투사체의 색 역할을 문서화한다.
- 모든 UI 패널은 같은 라운드, 같은 테두리, 같은 여백 규칙을 사용한다.
- 별/달 배경은 메뉴에는 어울리지만, 인게임에서는 스테이지별 바닥 질감과 위험 가독성이 우선이다.

즉시 개선하면 좋은 화면:

1. 메인 메뉴
   - 버튼 수가 많아 첫 화면이 설정 패널처럼 보인다.
   - `시작`, `강화`, `캐릭터`, `스테이지`를 1차 행동으로 묶고, `언어`, `크레딧`, `순위표`, `CODEX`는 보조 메뉴로 낮춘다.

2. 인게임 HUD
   - HP/XP 바가 너무 큰 상단 면적을 차지한다.
   - 시간, 레벨, 골드, 처치 수의 위계는 좋지만 레이아웃이 다소 정적이다.
   - 모바일에서는 손가락과 조이스틱이 하단을 가리므로 하단 UI는 최소화한다.

3. 레벨업 화면
   - 선택 카드가 명확한 것은 장점이다.
   - 카드 높이와 버튼 비중이 커서 "선택지 비교"보다 "버튼 누르기"가 먼저 보인다.
   - 무기 아이콘, 현재 레벨, 다음 변화, 빌드 태그를 더 작고 촘촘하게 넣는 편이 좋다.

4. 결과 화면
   - 보상과 업적이 잘 보이지만 축하감이 약하다.
   - 승리/패배에 맞는 색, 짧은 애니메이션, 획득 골드 카운트업, 신규 해금 강조가 필요하다.

### 4.2 UI 정보 위계

상용화용 UI 원칙:

- 1초 안에 "무엇을 누르면 되는지" 보여준다.
- 3초 안에 "이번 판에서 무엇이 달라졌는지" 보여준다.
- 10초 안에 "다음 목표가 무엇인지" 보여준다.

구체 작업:

- 메인 메뉴에 "오늘의 목표" 또는 "다음 해금까지 N 골드" 표시
- 상점에 강화 전/후 수치와 다음 비용 명확화
- 캐릭터 선택에 시작 무기 아이콘과 난이도 태그 추가
- 스테이지 선택에 예상 플레이 시간, 적 성향, 보상 배율 표시
- 레벨업 카드에 "신규", "강화", "진화 가능", "생존", "화력", "편의" 태그 추가

## 5. 게임 디자인 개선 방향

### 5.1 첫 3분 리듬

Survivors-like 장르는 "이동으로 적 웨이브를 처리하고 공격/패시브를 주기적으로 성장시키는 구조"가 핵심이다. 장르가 이미 널리 알려진 만큼, 평범한 자동 공격만으로는 차별화가 약하다.

현재 개선 방향:

- 0:00-0:20: 적을 충분히 보여주고 첫 처치/첫 보석을 빠르게 준다.
- 0:20-0:40: 첫 레벨업을 확실히 발생시킨다.
- 1:00: 첫 패턴 적 또는 미니 이벤트를 보여준다.
- 2:00: 플레이어가 첫 빌드 방향을 느끼게 한다.
- 3:00: "이번 판은 이런 판"이라는 인상이 생겨야 한다.

추가 권장:

- 첫 레벨업 선택지는 완전 랜덤보다 "공격 1 / 생존 1 / 편의 1"로 보장한다.
- 첫 60초에는 화면에 보여줄 적 종류를 2종으로 제한한다.
- 첫 미니보스는 단순 체력몹보다 작은 패턴 하나를 가져야 한다.

### 5.2 무기와 진화

현재 무기 5종과 진화 2종은 좋은 출발점이다. 상용화를 위해서는 각 무기가 숫자 차이가 아니라 "플레이 방식 차이"로 기억되어야 한다.

권장 역할:

| 무기 | 상용화 기준 역할 |
|---|---|
| Moon Dagger | 가장 믿을 수 있는 단일 타깃 추적, 보스 대응 |
| Spirit Orb | 근접 방어선, 안전지대 형성 |
| Fire Wisp | 랜덤 폭발이 아니라 위험 밀집 지역 정리 |
| Thorn Ring | 포위 돌파, 넉백과 생존 |
| Star Needle | 방향성 탄막, 넓은 청소 |

개선 작업:

- `Fire Wisp`는 랜덤 위치보다 적 밀집도 기반 타깃팅으로 바꾼다.
- `Star Needle`은 무작위 발사보다 플레이어 이동 방향 또는 적 밀집 방향과 연계한다.
- `Thorn Ring`은 타격 순간에 적 밀림과 링 잔상을 더 명확히 보여준다.
- 진화는 최소 5종 모두에게 제공하되, 한 번에 다 만들지 말고 1개씩 품질 기준을 맞춘다.
- 레벨업 설명은 "DMG +25% / CD -12%" 같은 공통 문구를 줄이고 실제 변화 문구를 보여준다.

### 5.3 적과 보스

적 종류는 이미 충분하다. 다음 과제는 "역할이 플레이어에게 보이는가"다.

권장:

- 돌진형은 돌진 전 텔레그래프 색/방향을 0.5초 이상 보여준다.
- 캐스터 투사체는 플레이어 투사체와 색/두께를 완전히 다르게 한다.
- Splitter는 분열 순간을 보상처럼 읽히게 하지 말고 위험 증가로 읽히게 한다.
- 미니보스는 HP 바만 추가하는 것이 아니라 등장 전 경고, 고유 색, 작은 패턴을 가진다.
- 최종 보스는 5분 생존의 장식이 아니라 마지막 30초 목표가 되어야 한다.

### 5.4 메타 성장

현재 골드, 영구 강화, 캐릭터/스테이지 해금이 있다. 상용화에서는 "다음 판을 해야 하는 이유"를 더 선명하게 만들어야 한다.

권장:

- 결과 화면에 "다음 강화까지 N 골드"를 표시한다.
- 스테이지 클리어 시 다음 스테이지 해금 조건을 바로 보여준다.
- 업적 보상은 획득 즉시 연출하고 상점으로 자연스럽게 연결한다.
- 강화는 10레벨까지 단순 수치 증가만 반복하지 말고 3/6/10 레벨에 작은 품질 변화를 준다.
- 과금 기능은 MVP 정책상 넣지 않는다. 광고/IAP도 현재 단계에서는 보류가 맞다.

Google Play 광고 정책상 보상형 광고처럼 사용자가 명시적으로 선택한 광고는 가능하지만, 게임 시작 직후나 플레이 중 예기치 않게 뜨는 광고는 사용자 경험을 해칠 수 있다. 이 프로젝트 정책과도 충돌하므로, 상용화 1차 버전은 유료/광고 없는 프리미엄 또는 무료 데모 + 후속 유료판 쪽이 더 안전하다.

## 6. 기술 개선 방향

### 6.1 데이터화

현재 `stages.json`은 좋은 방향이다. 같은 원칙을 확장해야 한다.

우선 분리할 데이터:

- `weapons.json`: 레벨별 피해량, 쿨다운, 투사체 수, 관통, 설명 키
- `passives.json`: 레벨별 효과, 상점/런타임 효과 구분
- `characters.json`: HP, 이동속도, 시작 무기, 해금 비용
- `enemies.json`: HP, 속도, 피해량, XP, 골드, 스프라이트
- `achievements.json`: 조건, 보상, 표시 문구

효과:

- 밸런스 패치가 GDScript 수정 없이 가능하다.
- 문서와 코드의 수치 불일치가 줄어든다.
- 출시 전 QA에서 CSV/JSON 비교로 변경점을 빠르게 볼 수 있다.

### 6.2 성능

현재 `EnemySpawner`와 무기 일부가 `get_tree().get_nodes_in_group()` 탐색을 자주 사용한다. 규모가 커지면 모바일에서 병목이 될 수 있다.

권장:

- 적 목록을 `EnemySpawner` 또는 `GameRoot`에서 등록/해제하여 캐시한다.
- 가장 가까운 적 탐색은 매 프레임이 아니라 0.1-0.2초 간격으로 갱신한다.
- 투사체, XP 보석, 골드 코인은 object pool을 적용한다.
- 데미지/폭발 VFX는 노드 생성/삭제보다 재사용 가능한 풀로 옮긴다.
- Android 중급기 기준 30 FPS 방어를 QA 기준으로 둔다.

Google Android 게임 문서도 프레임 페이싱과 성능 측정을 별도 주제로 다룬다. 상용화 단계에서는 "내 폰에서 잘 됨"이 아니라 프레임/메모리 기준을 정해 확인해야 한다.

### 6.3 저장 데이터

현재 저장 데이터에 version 필드가 없다. 상용화 후 저장 구조를 바꾸면 마이그레이션 위험이 커진다.

권장:

- 저장 파일에 `schema_version` 추가
- 신규 필드는 기본값으로 안전하게 보강
- 손상된 저장 파일 대응: 백업, 기본값 복구, 사용자 안내
- 테스트 저장 파일 세트 유지: 신규 유저, 중간 진행, 모든 해금, 손상 파일

이 변경은 저장 데이터 구조 변경이므로 구현 시 `.agent/decisions.md`에 마이그레이션 계획을 남겨야 한다.

### 6.4 플랫폼 기능

Google Play Games Level Up 가이드라인은 보상 여정, 플레이어 연속성, 크로스 디바이스 플레이를 강조한다. 특히 2026년 일정상 업적, Sidekick, Cloud Save가 중요해진다.

현재 방향:

- 업적과 리더보드는 이미 좋은 출발점이다.
- Cloud Save는 현재 MVP 정책의 "온라인 저장 제외"와 충돌하므로, 상용화 정책을 별도로 결정해야 한다.
- PGS가 선택 기능인지 필수인지, 오프라인 플레이가 완전히 가능한지 명확히 해야 한다.

권장:

- 1차 상용화는 오프라인 완전 플레이 유지
- PGS 로그인 실패 시 메뉴/게임 진행이 막히지 않도록 QA
- 리더보드 점수 공식과 제출 조건을 문서화
- Play Console의 리더보드 통계로 점수 제출률을 추적

## 7. 출시 전 QA 기준

### 기능 QA

- 첫 실행에서 언어 자동 감지와 폰트 출력 확인
- 캐릭터/스테이지/난이도 선택 후 실행 확인
- 5개 무기 획득, 레벨업, 진화 확인
- 모든 패시브 적용 확인
- 보스 등장, 보스 처치, 시간 클리어 확인
- 사망 결과, 승리 결과, 골드 저장 확인
- 상점 구매, 캐릭터 해금, 스테이지 해금 확인
- 앱 종료 후 저장 유지 확인
- PGS 미지원 환경에서 메뉴가 깨지지 않는지 확인

### 밸런스 QA

기본 지표:

- 첫 레벨업 평균 시간: 20-35초
- 첫 사망 중앙값: Normal 기준 3분 이후
- 첫 클리어까지 필요한 판 수: 3-8판
- Normal 클리어율: 숙련 전 20-40%, 숙련 후 60% 이상
- 한 판 평균 골드: 다음 강화까지 너무 멀지 않게 조정
- 보스전 사망 비중: 너무 높으면 보스보다 직전 웨이브를 조정

### UX QA

- 한 손 플레이 시 조이스틱이 전투 판단을 가리지 않는지
- 레벨업 카드가 작은 화면에서도 누르기 쉬운지
- 한국어/영어 문구가 버튼 밖으로 넘치지 않는지
- 색만으로 위험을 구분하지 않아도 되는지
- 5분 플레이 후 눈 피로가 심하지 않은지

### 스토어 QA

- 스토어 스크린샷이 현재 게임 화면과 일치하는지
- 개인정보 처리방침과 실제 데이터 수집이 일치하는지
- 광고/IAP 없음 여부가 스토어 표시와 일치하는지
- PGS, 리더보드, 업적 설명이 실제 동작과 일치하는지
- APK/AAB, Windows, Web 빌드가 모두 같은 버전 표기를 갖는지

## 10. 최종 판단

`Nightseed Survivor`는 "만들 수 있을까?" 단계는 이미 넘었다. 이제 핵심 질문은 "낯선 사람이 스토어에서 보고 설치한 뒤, 첫 판이 끝나고 한 번 더 누를까?"다.

상용화의 첫 관문은 콘텐츠 추가가 아니라 신뢰감이다. 화면이 정돈되어 있고, 조작이 편하고, 선택지가 이해되고, 죽어도 다음 목표가 보이면 작은 게임이어도 제품처럼 느껴진다.

따라서 다음 작업 루프의 추천 시작점은 디자인 리워크다. 특히 메인 메뉴, HUD, 레벨업 카드, 결과 화면을 먼저 다듬으면 현재 구현된 전투/성장/스토리 기능이 훨씬 더 좋아 보이고, 이후 밸런스 튜닝의 효과도 사용자에게 더 잘 전달된다.
